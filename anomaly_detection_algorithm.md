# 維護事件異常判定算法 v1.0

## 核心原理

根據 ISO 19030 Speed Loss 分析框架，通過前後 45 天中位數對比判定維護效果：

```
Delta = After_Median_SL - Before_Median_SL

Delta < 0：改善（Speed Loss 降低，性能好）
Delta > 0：未改善或惡化（Speed Loss 增加，性能差）
```

## 按維護類型的異常判定標準

### 1. DD（進塢）- 最高效維修
- **期望改善範圍**：-20% ~ -5%（顯著改善）
- **異常判定**：Δ < -20% 或 Δ > -3%
- **樣本統計**：n=7, 改善 85.7%, σ=16.03%
- **解釋**：
  - Δ < -20%：改善超常（新塗層初期最優化）
  - Δ > -3%：改善不足（可能施工不完善或海況影響）

### 2. UWC+PP / UWI+PP（複合維修）
- **期望改善範圍**：-15% ~ -5%
- **異常判定**：Δ > -2.5%
- **樣本統計**：n=38, 改善 54%, σ=17.5%
- **解釋**：清洗 + 拋光，應有明顯效果

### 3. PP（螺旋槳拋光）/ UWC（船殼清洗）
- **期望改善範圍**：-10% ~ -3%
- **異常判定**：Δ > -1.5%
- **樣本統計**：n=17, 改善 63%, σ=15.9%
- **解釋**：單項維修，效果有限

### 4. UWI（純水下檢查）
- **期望改善範圍**：≈ 0% （不應有改善）
- **異常判定**：Δ > 1.0%
- **樣本統計**：n=12, 改善 58% ⚠️, σ=18.84%
- **解釋**：
  - 純檢查無物理介入，不應改善
  - 任何改善 > 1% 都標記為異常，說明檢查報告與實際不符

## 多維異常檢測（擴展）

### 建議加入的特徵

1. **風浪條件**
   - `WIND_SCALE`：維護前後的海況變化可能掩蓋真實效果
   - 海況好的時期通常 SL 較低，但與維修無關

2. **船舶負載**
   - `CARGO_ON_BOARD`、`DISPLACEMENT`：吃水深度影響阻力
   - 維修前後若吃水變化 > 2m，需要調整期望值

3. **航行工況**
   - `ME_AVG_RPM`、`LOAD_PCT`：不同轉速的維修效果不同
   - 低速運行時清洗效果可能被掩蓋

4. **燃油消耗（FOC 偏離）**
   - 結合 SFOC 或 ME_CONSUMPTION 數據
   - 如果 SL 未改善但油耗反而降低，可能是主機效率問題而非船殼

## 實施建議

### 前端異常標記規則

```javascript
function isAnomalous(event) {
  const { delta, event_type } = event;
  
  const thresholds = {
    'DD': { min: -20, max: -3 },
    'UWC+PP': { min: -15, max: -2.5 },
    'UWI+PP': { min: -15, max: -2.5 },
    'PP': { min: -10, max: -1.5 },
    'UWC': { min: -10, max: -1.5 },
    'UWI': { min: -10, max: 1 }  // 純檢查不應改善
  };
  
  const t = thresholds[event_type];
  return delta < t.min || delta > t.max;
}
```

### 儀表板顯示

- 🟢 正常改善：Δ 在預期範圍內
- 🟡 邊界異常：Δ 略超出範圍（1-2%）
- 🔴 明確異常：Δ 顯著超出範圍
