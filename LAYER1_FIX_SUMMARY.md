# Layer 1 修複 — 從「全歷史最佳」到「進塚後基線」

## 🔴 原始問題

你指出：**維修完成後，Speed Loss 反而提升，這不合理**

### 根本原因

舊的 Layer 1 邏輯使用 **全歷史 5 年的 top 10% STW** 作為 baseline：

```
Baseline = max(STW) in history (2021-2025)
Speed Loss = (Baseline - Actual STW) / Baseline × 100

結果：
  ✗ Baseline 是不現實的「歷史最高峰」
  ✗ 進塚後的船體也達不到 5 年前最乾淨時的速度
  ✗ 維修效果被隱沒，變成負值 60-80%
  ✗ 物理意義錯誤：測的是「與歷史最高的差距」，而非「汙損程度」
```

---

## ✅ 修複方案

改為 **進塚後 7-14 天的 top 10% STW** 作為 baseline：

```python
# 新邏輯
Baseline = 90th percentile(STW in days [maint_day+7, maint_day+14])
Speed Loss = (Baseline_clean - Actual STW) / Baseline_clean × 100

優點：
  ✓ Baseline = 實際達到過的「清潔狀態」
  ✓ 後續衰退 = 真正的「汙損程度」
  ✓ 維修後 SL 明顯下降
  ✓ 物理意義正確
```

## 📈 新的 Speed Loss 特性

### 時間序列特徵（現在正確）

```
維修前期：SL 逐漸上升（汙損累積）
  SL = 0% → 3% → 8% → 12% (汙損嚴重)

維修日 (DD/UWC)：基線重設
  Baseline = 進塚後最乾淨的 10%

維修後期：SL 從低點重新上升
  SL = 0% → 2% → 6% (新汙損周期)

結果：
  ✓ 鋸齒波形（sawtooth pattern）
  ✓ 維修點處明顯下降
  ✓ 物理可解釋
```

## 📝 技術細節

### Baseline 選擇邏輯

```
IF cycle_num == 0 (第一個週期，維修前)
  → 使用該週期的全部數據（best effort）

ELSE (維修後的週期)
  → 查找 [maint_day+7, maint_day+14] 範圍內的數據
  → 如果不足，回退到 [maint_day, maint_day+14]
  → 按 RPM 分組，計算每組的 90th percentile STW
  → 作為該週期的 baseline
```

### RPM 正規化的額外意義

現在 RPM 正規化改善更準確，因為：
1. Baseline 是實際達到的清潔狀態
2. 不同 RPM 帶有獨立的 baseline
3. 同 RPM 下的改善 = 純淨的除污效果

## ⚠️ 已知限制

1. **週期邊界附近的數據少**
   - 維修事件在數據邊界時，可能沒有 7-14 天的完整數據
   - 77 個維修事件中只有 13 個有完整前後數據

2. **第一個週期（cycle 0）**
   - 使用全周期數據，不精確
   - 但通常 cycle 0 數據量多，相對可靠

3. **季節性偏差**
   - 不同季節的最佳 STW 可能不同
   - 目前未分季節，可作為未來改進
