# 陽明海運 Speed Loss 分析系統
## AWS Summit Hackathon 2026

### 📌 項目概述

本項目實現了一套**完整的船舶推進效能分析系統**，基於 ISO 19030 標準，通過分層計算方法量化船舶 Speed Loss（速度損失），並提供可視化儀表板支援決策。

**核心功能：**
- ✅ 自動化數據清洗與聚合（21,000+ 航日記錄）
- ✅ 多層次 Speed Loss 計算（3 種方法）
- ✅ 維修效益量化與追蹤
- ✅ 異常檢測與告警系統
- ✅ 交互式可視化儀表板
- ✅ 自動報告生成

## 📊 Speed Loss 計算方法

### 層級 1：相對基準法
**公式：**
```
Speed Loss L1 = (Daily FOC - 該船歷史最佳 FOC) / 基準 FOC × 100%
```

**特點：**
- 直觀易懂
- 無需複雜建模
- 自動納入船齡等因素

**應用：** 快速效能趨勢識別

---

### 層級 2：多因素回歸法
**公式：**
```
Expected FOC = Baseline × (1 + 衰退因子 + 風力因子 + 季節因子 + 負載因子)
Speed Loss L2 = (Expected - Actual) / Expected × 100%
```

**分解因子：**
- **衰退因子**（15%）：設備隨時間推移的性能損耗
- **風力因子**（10%）：環境條件的影響
- **季節因子**（±5%）：季節性海況變化
- **負載因子**（5%）：貨物負載效應

**特點：**
- 控制環境變量
- 隔離設備狀態
- 支持預測維修時機

**應用：** 精準的衰退趨勢分析

---

### 層級 3：維修效益法
**公式：**
```
效益 = (維修前平均 FOC - 維修後平均 FOC) / 維修前 FOC × 100%
```

**關鍵事件：**
- UWI（水下檢查）
- PP（螺旋槳拋光）
- UWC（船殼清洗）
- DD（進塢保養）

**應用：** 量化 ROI，支援維修決策

## 📈 關鍵指標說明

### 1. 船隊平均 Speed Loss
- **目標：** < 10%
- **良好：** 10-15%
- **警告：** 15-20%
- **臨界：** > 20%

### 2. 單船表現評級
| Speed Loss | 評級 | 建議行動 |
|-----------|------|---------|
| < 8% | 優秀 | 保持現狀 |
| 8-12% | 良好 | 常規監控 |
| 12-18% | 需關注 | 檢查維修狀態 |
| > 18% | 緊急 | 立即安排檢查 |

### 3. 維修效益
- **優秀：** > 15%
- **良好：** 10-15%
- **中等：** 5-10%
- **差：** < 5%

### 場景 1：日常監控（周）
**任務：** 檢查是否有異常船舶
**流程：**
1. 執行 `speed_loss_pipeline.py`
2. 查看 `anomalies.csv`

### 場景 2：維修計劃制定（月）
**任務：** 規劃下月維修安排
**流程：**
1. 分析 Speed Loss L2 的衰退曲線
2. 對比歷史維修效益數據
3. 計算最佳維修時機

### 場景 3：船隊績效評估（季）
**任務：** 評估船隊整體表現
**流程：**
1. 生成季度報告
2. 對比同型船性能
3. 識別異常表現船舶

**決策：** 資源分配與激勵計劃

---

### 場景 4：根因分析（即時）
**任務：** 調查特定異常事件
**流程：**
1. 從異常告警開始
2. 整合多維數據：天氣、海況、維修、負載
3. 與船員溝通驗證

**決策：** 根本原因與對策

## 🐛 常見問題

**Q: Speed Loss 為什麼這麼高？**  
A: 可能原因包括設備衰退、需要維修、不利的航行條件或數據異常。使用 L2 分析幫助排除環境干擾。

**Q: 多久運行一次計算？**  
A: 推薦週期為每週一次日常監控，每月一次統計分析，維修後立即驗證。

**Q: 如何解釋異常的 Speed Loss 值？**  
A: 異常低值可能表示設備超常表現（稀有），異常高值表示嚴重衰退或數據質量問題。
