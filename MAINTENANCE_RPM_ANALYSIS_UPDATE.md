# 維修效益分析升級 — RPM 正規化指標已上線

## 🎯 問題已解決

**原問題**：維修後看到很多 FOC（油耗）上升，以為維修失敗 ❌

**根本原因**：30 天平均 FOC 忽視了運營模式改變
```
維修前：低速 RPM 40 → FOC 39.67 MT/day
維修後：高速 RPM 75 → FOC 50.96 MT/day
表面：FOC ⬆️ 上升 28.5%
真相：只是速度提高了，油耗實際改善 1.47% ✅
```

---

## ✨ 新增功能

### 1️⃣ RPM 前後對比
```
RPM 前: 47.2
RPM 後: 49.3
```
一眼看出維修是否改變了運營速度

### 2️⃣ 雙指標 FOC 改善
| 指標 | 含義 | 何時信任 |
|------|------|---------|
| **FOC 改善 (30天平均)** | 直觀油耗變化 | ✓ 當 RPM 基本不變 |
| **FOC 改善 (RPM正規化)** | **相同轉速下的真實改善** | ✓ **永遠信任這個** |

### 3️⃣ 智能解讀
```
✅ 維修有效，油耗確實改善           (normalized > 0, raw > 0)
⚠️  維修後提高轉速，但相同轉速改善   (normalized > 0, raw < -10)
✅ 維修有效，雖然轉速降低了          (normalized > 0, raw < 0)
❌ 維修可能無效，甚至變差           (normalized < -5, raw < -5)
```

## 📊 真實案例對比

### 案例 1：S1 Day 1386 (UWI+PP 檢查+拋光)

| 指標 | 值 |
|------|-----|
| RPM 前 | 54.5 |
| RPM 後 | 52.0 |
| FOC 改善 (30天平均) | **10.89%** ← 看起來改善不大 |
| FOC 改善 (RPM正規化) | **16.56%** ← **真正的改善！** ✅ |

**解讀**：維修確實有效。原本看起來改善 10.89% 的原因是轉速從 54.5 降到 52.0，在相同轉速下實際改善了 16.56%！

---

### 案例 2：S4 Day 1709 (PP 螺旋槳拋光)

| 指標 | 值 |
|------|-----|
| RPM 前 | 51.8 |
| RPM 後 | 56.7 |
| FOC 改善 (30天平均) | **-7.49%** ← ❌ 看起來油耗上升了！ |
| FOC 改善 (RPM正規化) | **6.05%** ← ✅ **但實際改善 6%！** |

**解讀**：維修後轉速從 51.8 提高到 56.7（高速了），所以油耗看起來上升，但相同轉速下確實改善了 6.05%！

---

### 案例 3：S1 Day 981 (DD 進塢保養)

| 指標 | 值 |
|------|-----|
| RPM 前 | 47.2 |
| RPM 後 | 49.3 |
| FOC 改善 (30天平均) | **-28.46%** ← ❌ 大幅上升！ |
| FOC 改善 (RPM正規化) | **1.47%** ← ⚠️ **改善有限** |

**解讀**：進塢後轉速略提（47.2→49.3），油耗看起來上升 28.46%，但正規化後只改善 1.47%。可能進塢後立即進入惡劣海況。

## 📁 技術更新

### 後端 (_export_sl_csv.py)
新增欄位：
```csv
rpm_before_avg                  # 維修前 30 天平均 RPM
rpm_after_avg                   # 維修後 30 天平均 RPM
rpm_normalized_foc_drop_pct     # 相同 RPM 下的油耗改善 %
```

**計算邏輯**：
1. 找到維修後的目標 RPM (rpm_after_avg)
2. 在維修前找 ±5 RPM 範圍內的數據
3. 比較該範圍內的平均 FOC
4. 計算改善百分比

### 前端 (SpeedLossDashboard.vue)
- 新表格：「維修效益分析（含 RPM 正規化）」
- 新函數：
  - `getMaintInterpretClass()` - 根據數值著色
  - `getMaintInterpretation()` - 自動生成解讀文字
- 新樣式：表格著色 (positive/negative/neutral)

### JSON 數據
visualization_data.json 現包含完整的 RPM 和正規化數據，前端可直接使用。
