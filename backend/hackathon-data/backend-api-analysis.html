一、必要的 Backend API List（完整清單 + 算法說明）

根據題目要求（油耗預測模型 + Speed Loss Dashboard）及前端頁面需求，以下是完整的後端 API 清單：

1.1 核心 API（直接對應評分項目 55%）

APIMethod & Path功能說明算法/實作方式
油耗預測
評分25%		 POST /api/v1/predict/fuel-consumption 給定船舶 + 航行條件，預測主機全速油耗(MT/day)。支援反事實推論：「執行 UWC/PP 後油耗會降多少」 算法：
1. 特徵工程：從 vt_fd.csv 提取 A 類特徵 + 時間退化特徵（距上次養護天數、養護類型 one-hot）
2. 模型：XGBoost/LightGBM 回歸（或 hybrid：物理基線 + ML 殘差修正）
3. 訓練：用 S1-S12 完整數據訓練，篩選全速≥22hr、風速≤4 的穩態日為主要訓練樣本
4. 反事實：將「距上次養護天數」重設為 0 或指定天數重新預測，差值即節能效益
5. 輸出：predicted_value (MT/day), confidence_interval
Speed Loss 計算
評分30%		 GET /api/v1/vessels/{vessel}/speed-loss ISO 19030 框架下計算 Speed Loss 時間序列 算法：
1. 基線建立：取該船最早期（或坐塢後）乾淨狀態的 Power-Speed 曲線作為 reference
2. Speed Loss % = (V_ref - V_actual) / V_ref × 100，在相同功率/排水量條件下比較
3. 功率修正：用 Admiralty Coefficient 或多項式擬合 Power = f(Speed, Displacement, Draft)
4. 環境修正：ISO 15016 方法扣除風浪影響（Wind Resistance、Added Wave Resistance）
5. 回傳：每日 speed_loss_pct 時間序列 + 養護事件標記

1.2 Dashboard 支援 API（對應評分 Speed Loss Dashboard 30%）

APIMethod & Path功能說明算法/實作方式
Speed Loss 歸因 GET /api/v1/vessels/{vessel}/speed-loss-attribution Speed Loss 中有多少 % 歸因於船殼 vs 螺旋槳 算法：
1. 利用 maintenance.csv 中 PP（僅螺旋槳拋光）事件前後的 speed loss 變化量 → 螺旋槳貢獻
2. UWC（僅船殼清洗）事件前後的 speed loss 變化量 → 船殼貢獻
3. UWC+PP 事件的恢復量 - PP only 恢復量 = 船殼貢獻（差值法）
4. 比例 = hull_contribution / total_speed_loss × 100%
5. 關鍵：UWI（純檢查）不應帶來效能改善，用作控制組驗證
養護事件效益對比 GET /api/v1/vessels/{vessel}/maintenance-effectiveness Speed Loss 與養護事件的時序對應（恢復/惡化時間點） 算法：
1. 對每個養護事件，取事件前 7-14 天的 avg speed_loss vs 事件後 7-14 天的 avg speed_loss
2. recovery_pct = before - after（正值=恢復，負值=惡化或無效）
3. 依事件類型（DD/UWC/PP/UWI）分組統計平均恢復量
4. UWI 的 recovery 應接近 0 或隨機波動（驗證模型合理性）
養護事件列表 GET /api/v1/vessels/{vessel}/maintenance-events 回傳該船所有養護事件（含類型、日期、狀態描述）供圖表標註 直接讀取 maintenance.csv 篩選該船，格式化回傳

1.3 商務決策 API（對應評分 商務決策價值 20%）

APIMethod & Path功能說明算法/實作方式
節能效益模擬 POST /api/v1/vessels/{vessel}/energy-saving-simulation 模擬「現在執行 UWC/PP」vs「延遲 N 天」的累積油耗差異成本 算法：
1. 用油耗預測模型的反事實推論能力
2. 計算 current speed_loss → 日均超額油耗 (MT)
3. 預測養護後 speed_loss 恢復值 → 養護後日均油耗
4. 差值 × 燃料單價 × 延遲天數 = 累積損失
5. 與養護成本比較 → breakeven_days
維修建議 GET /api/v1/vessels/{vessel}/maintenance-recommendation 給出最佳維修時機建議、急迫度、預計節省金額 基於退化速率 + 歷史清潔效益 + 成本模型，計算最佳介入時間點
船隊排名/比較 GET /api/v1/fleet/ranking 跨船隊 Speed Loss/油耗效率排名，識別優先處理船舶 計算各船當前 speed_loss、退化速率、距上次養護天數，加權排序

1.4 數據支援 API

APIMethod & Path功能說明算法/實作方式
船舶列表 GET /api/v1/vessels 回傳所有船舶基本資訊 + 當前狀態摘要 讀取 vt_fd.csv 提取各船最新一筆數據 + maintenance.csv 計算衍生指標
Noon Report 列表 GET /api/v1/vessels/{vessel}/noon-reports 回傳該船所有航行日報原始數據 直接讀取 vt_fd.csv 篩選該船，回傳所有 A 類欄位
油耗歸因分解 GET /api/v1/vessels/{vessel}/fuel-attribution 當前油耗的因子分解（天氣/船殼/螺旋槳/主機老化） 算法：使用 SHAP 或特徵重要性分解模型預測結果中各因子的貢獻量
異常偵測 GET /api/v1/vessels/{vessel}/anomalies 偵測異常高油耗或異常 speed loss 的日子 Z-score 統計方法 + 規則判定（如突然退化可能是附著生物爆發）
預測結果提交 POST /api/v1/predict/submit 產出 102 筆預測結果 CSV，供評審評分 對 S21-S23 的 PREDICT 標記日逐一呼叫預測模型，組裝成提交格式

3.5 關鍵技術提醒

⚠️ 題目陷阱 — UWI（水下檢查）不應帶來效能改善
題目明確提示：「有一類事件屬於純檢查、無實體介入——它不清洗、也不拋光。聰明的模型應能辨識出這類事件不應帶來效能改善。」

實作建議：
• 在 maintenance.csv 的 event_type 中，UWI = Underwater Inspection（僅拍照）
• 建模時 不要 將 UWI 視為「養護事件」重設退化計時器
• Dashboard 上可以用 UWI 事件前後的 speed loss 變化作為「控制組」展示模型合理性
• 這是評審會特別看的亮點
💡 預測模型要點
• 預測日條件：全速≥22hr、風速≤4、單一燃料 → 這些是乾淨穩態點
• 全速時數校正：題目提醒「船隻每日全速航行時長可能有異，數學模型應執行校正」
• 多燃料處理：不同日使用不同燃料（HSHFO/VLSFO/ULSFO/LSMGO），熱值不同需折算或分模型
• 退化特徵：days_since_dd、days_since_uwc、days_since_pp 是核心時間衰退特徵
• 船型分組：S1-S8,S21 為 W1 型；S9-S12,S22-S23 為 W2 型（同設計但不同航線）
✅ 加分亮點建議
• AI 協作：讓 Chatbot 可以回答「S21 上次養護是什麼時候？」「如果現在做 UWC 可以省多少油？」
• 商務決策：在 Dashboard 上直接顯示「如果現在執行 UWC，預計年節省 $XXX USD」
• 視覺化：Speed Loss 圖上標注不同類型養護事件（DD=綠、UWC=藍、PP=橙、UWI=灰虛線），讓評審一眼看出效果差異
• UWI 控制組：專門做一個小面板展示「UWI 前後 Speed Loss 無顯著變化」→ 證明模型的因果識別能力
