# VoiceBot / Chatbot 優先互動模式 — UX 需求文件

> 給 Kiro / Claude Code 的 UX 開發需求文件
> 版本：v1.0　最後更新：2026-07-10
> 前置文件：《船隊維運 Dashboard — Frontend Design Requirements》（本文件為其擴充，共用同一套 design tokens）

---

## 1. 概述與目標

在現有的 Dashboard（滑鼠點擊操作）之外，新增**第二種對等的互動模式**：以語音或文字輸入為主要操作方式，取代點擊瀏覽，讓使用者用「問問題」的方式取得船隊維運資訊。

**核心原則：兩種模式是並存的，不是取代關係。**
- Dashboard 模式：適合深入分析、比較、細看圖表細節
- VoiceBot 模式：適合快速取得結論、免手操作情境（例如在船舶巡檢現場、會議中口頭詢問）、對不熟悉介面的高層使用者更友善

使用者應該可以**隨時在兩種模式間切換**，且切換後上下文（目前正在看哪艘船、哪個時間區間）要能延續，不必重新輸入。

---

## 2. 使用情境（why this matters）

| 情境 | 說明 |
|---|---|
| 高層快速決策 | Fleet Manager 開會時想知道「哪三艘船最需要優先安排維修」，用語音問比翻找 Dashboard 快 |
| 現場巡檢 | Superintendent 在船上用手機，雙手可能不便操作觸控，語音輸入更實際 |
| 新手上手 | 不熟悉船隊分析術語與介面結構的使用者，用自然語言提問比學習介面架構門檻更低 |
| 交班/交接 | 快速用一句話取得「這艘船目前狀態」的總結,不需重新熟悉儀表板佈局 |

## 3. 互動模式架構

```
┌─────────────────────────────────────────┐
│           全站頂部：模式切換器              │
│   [ Dashboard 模式 ]  [ 對話模式 ]          │
└─────────────────────────────────────────┘

Dashboard 模式（既有）：
  滑鼠點擊 → 逐頁瀏覽 → 圖表互動

對話模式（新增）：
  語音/文字輸入 → 意圖解析 → 畫面區塊逐一淡入呈現
  → 使用者可繼續追問 → 或一鍵「查看完整 Dashboard」跳轉回對應頁面
```

**關鍵行為**：對話模式**不是取代 Dashboard 頁面**，而是「用對話方式召喚 Dashboard 裡既有的資料區塊」。也就是說,底層資料元件（測深計量表、Speed Loss 圖表、歸因瀑布圖）完全重複使用第一份需求文件裡定義的元件,只是呈現的觸發方式從「點擊導航」變成「語音/文字指令」。

## 4. VoiceBot/Chatbot 視覺設計要求

**明確要求：不要做成通用的右下角聊天泡泡 widget。** 這個介面本身要有足夠的設計感，是產品的第二個核心體驗，需要延續第一份文件第 3 節的「航海圖 × 船舶儀表」視覺語言，而非套用一般 SaaS 客服機器人的樣板（圓形頭像+氣泡對話框+打字動畫三件套）。

### 4.1 視覺定位建議：「聲測儀 / 通訊台」意象
以船舶通訊室或聲測儀操作台為靈感,而非聊天 App：

- **輸入區**不是一般聊天輸入框，設計成類似「艦橋通訊面板」：一條橫向的輸入軌道，語音輸入時中央顯示**波形視覺化**（呼應第一份文件的 signature 測深計元件語言，用同一組 `--brass-amber` / `--fathom-teal` 配色）
- **回應呈現**不是對話氣泡堆疊，而是**畫面區塊被「喚出」的過程本身就是回應**——AI 的文字簡答可以精簡地顯示在輸入軌道上方一行，但主要回應是畫面上的資料區塊逐一淡入（見第 5 節）
- 若做語音喚醒/聆聽中狀態，用**脈動的聲波環**（同心圓漸淡擴散,像聲測儀發出的訊號),而非常見的跳動麥克風圖示

### 4.2 全站定位
- 建議做成**可召喚的全螢幕疊層（overlay）**，而非固定側邊小視窗：按下切換鍵或快捷鍵（如 `Cmd/Ctrl + K` 或麥克風圖示），從畫面中央或上方展開，背景 Dashboard 微微變暗/模糊，聚焦感更強，體驗上更接近「切換操作模式」而非「開一個小工具」
- 關閉/返回 Dashboard 的方式要明確：ESC 鍵、右上角明確的「返回儀表板」按鈕、或說出「回到儀表板」皆可觸發

### 4.3 動態效果原則
- 語音波形動畫可以是全站唯一允許「持續性」動畫的地方（呼應第一份文件「把大膽留給一個地方」的原則）
- 畫面區塊淡入動畫需節制、有節奏感,不能同時全部跳出（見第 5 節細節）
- 尊重 `prefers-reduced-motion`，降級為無動畫的即時切換

## 5. 核心互動流程：指令 → 畫面逐一淡入

這是本文件最重要的互動規格。

### 5.1 流程
1. 使用者輸入語音或文字指令（例如：「MV EXAMPLE 現在狀況怎麼樣？」）
2. 系統顯示「理解中」的簡短過渡態（波形收斂或簡短 loading 提示，**不超過 1-2 秒**）
3. 系統給出**一句話文字結論**（例如：「MV EXAMPLE 目前 Speed Loss 8.2%，高於船隊平均，建議 3 週內安排水下清洗。」）
4. 對應的資料區塊**依重要性順序、逐一淡入**顯示在下方畫面（非同時出現）：
   - 第一個淡入：測深計風險量表（最關鍵的單一指標）
   - 第二個淡入：Speed Loss 迷你圖表
   - 第三個淡入：維修建議卡
   - 每個區塊淡入間隔建議 150-250ms，讓使用者視線有節奏地被引導,而非資訊轟炸
5. 使用者可以：
   - 針對淡入的區塊繼續追問（例如：「這個結論的信心程度多高？」→ 系統標註哪個區塊在回答,可用短暫的邊框強調 highlight）
   - 點擊任一淡入區塊右上角的「查看完整頁面」小按鈕，直接跳轉到 Dashboard 對應頁面並保留上下文
   - 說「回到儀表板」或按返回鍵，退出對話模式

### 5.2 淡入區塊的呈現規則
- 每次淡入的區塊**必須是既有 Dashboard 元件的精簡版**（同一套元件、資料來源，只是尺寸/資訊密度做卡片化簡化），確保視覺一致性,不要為對話模式重新設計一套獨立元件
- 若指令對應到多艘船（例如：「哪三艘船最需要優先維修？」），淡入的是**排行卡片列表**而非個別船的完整區塊
- 若資料不存在或不足，淡入的區塊要明確呈現「資料侃限」提示卡（呼應第一份文件維修建議頁的「未盡之處」設計），不可靜默省略

### 5.3 追問與多輪對話
- 系統需保留對話上下文（目前討論的是哪艘船、哪個時間範圍），使用者可以說「上個月呢？」而不必重複船名
- 畫面上應有簡短的**對話歷程軌跡**（不是完整聊天記錄堆疊，而是像「麵包屑」一樣的簡短標籤列，例如：`MV EXAMPLE › Speed Loss › 上月`），方便使用者知道目前對話上下文,也可點擊麵包屑回到之前的討論節點

## 6. 指令範例與對應畫面反應（供 Kiro 開發意圖對照表）

| 使用者輸入範例 | 意圖分類 | 淡入區塊順序 |
|---|---|---|
| 「MV EXAMPLE 現在狀況怎麼樣？」 | 單船總覽查詢 | 測深計風險量表 → Speed Loss 迷你圖 → 維修建議卡 |
| 「哪些船需要優先安排維修？」 | 跨船隊排行查詢 | 優先排行卡片列表 → 可展開查看個別船隻 |
| 「這艘船的油耗異常是船體還是螺旋槳造成的？」 | 歸因查詢 | 油耗歸因 Waterfall 圖精簡版 → 白話結論句 |
| 「幫我比較 A 船跟 B 船的污損趨勢」 | 跨船比較 | 疊圖比較卡片 |
| 「上次水下清洗是什麼時候？」 | 單一事實查詢 | 僅文字回答，不需淡入完整區塊（避免過度反應簡單問題） |
| 「這個建議的信心程度多高？」（追問） | 追問/延伸 | 針對前一個淡入區塊加註信心說明,不重新淡入整組區塊 |

**設計原則**：簡單的事實型問題只需文字回答，不必每次都觸發畫面動畫；只有需要視覺輔助理解的複雜查詢才觸發區塊淡入，避免「為了展示互動效果而過度動畫化」。

## 7. 語音輸出（TTS）設計考量

- 建議**預設僅語音輸入 + 文字/畫面輸出**，不強制語音回覆播報（船隊辦公室/會議情境中,語音一直講話可能干擾他人）
- 語音回覆可作為**可選開關**（例如在巡檢現場雙手不便看畫面時使用）
- 若啟用語音回覆，語氣需精簡（一句話結論,不要把整段文字逐字唸出），聲音調性建議沉穩、專業，避免過度活潑的預設 TTS 語調

---

## 8. 從對話模式返回 Dashboard 的機制

- 任何淡入的區塊皆有「查看完整頁面」按鈕，點擊後直接跳轉至 Dashboard 對應頁面（例如淡入的 Speed Loss 迷你圖 → 跳轉到 `/vessels/:imo/speed-loss` 並帶入相同的日期區間篩選）
- 全站固定可見的「返回儀表板」出口（快捷鍵 + 按鈕雙重保障）
- 兩種模式切換時**保留上下文**：從對話模式跳轉到 Dashboard，該頁面應直接呈現剛才討論的船隻/時間範圍，不需使用者重新篩選

## 9. 技術實作建議

| 項目 | 建議 |
|---|---|
| 語音輸入 (STT) | Web Speech API（瀏覽器原生,免費;若需更高辨識率可選 Whisper API） |
| 語音輸出 (TTS，選用) | 瀏覽器原生 SpeechSynthesis API 或第三方 TTS 服務(需可調語速/語氣) |
| 意圖解析 (NLU) | 用 LLM（Claude API）做意圖分類 + 參數萃取（船名/時間範圍/查詢類型），輸出結構化 JSON 再交給前端決定要淡入哪些既有元件，而非讓 LLM 直接生成畫面內容 |
| 前端動畫 | Vue `<Transition>` / `<TransitionGroup>` 搭配 stagger 延遲，控制區塊逐一淡入的節奏 |
| 上下文管理 | Pinia store 統一管理「目前對話上下文」（船隻/時間範圍/查詢歷程），供 Dashboard 與對話模式共用 |

**重要設計原則**：LLM 只負責「理解使用者想問什麼」並轉成結構化查詢參數，實際畫面呈現的仍是第一份需求文件裡定義好的既有元件與真實數據，確保呈現的資訊經過驗證、不是 LLM 生成的自由文字內容（避免油耗歸因等關鍵決策數字被模型幻覺誤植）。

## 10. 容錯與邊界情況設計

- **辨識失敗**：語音辨識信心過低時，明確顯示「沒聽清楚，請再說一次」，不要猜測後給出錯誤結論
- **查無資料**：指令對應的船隻/資料不存在時，回應要明確（「找不到叫 XXX 的船，你是不是指 MV EXAMPLE？」），提供修正建議而非空白畫面
- **超出範疇的問題**：若使用者問了系統無法回答的問題（例如天氣預報、與船隊資料無關），明確告知能力範圍,並列出 2-3 個範例問題引導使用者

## 11. 驗收檢查清單

- [ ] VoiceBot/Chatbot 視覺與 Dashboard 共用同一套 design tokens，不是通用聊天 widget 樣板
- [ ] 語音波形/聆聽中狀態有獨立設計，非跳動麥克風圖示
- [ ] 畫面區塊為既有 Dashboard 元件重用，非重新設計的獨立元件
- [ ] 區塊淡入為逐一節奏出現（150-250ms 間隔），非同時全部顯示
- [ ] 簡單事實型問題僅文字回答，不觸發整組區塊動畫
- [ ] 每個淡入區塊有「查看完整頁面」跳轉，且跳轉後保留上下文
- [ ] 對話歷程有簡短麵包屑軌跡，可回溯上下文
- [ ] 資料不足/查無資料時有明確引導，不靜默失敗
- [ ] 兩種模式間切換不遺失上下文（船隻/時間範圍）
- [ ] 尊重 `prefers-reduced-motion`
