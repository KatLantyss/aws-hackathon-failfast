## 0. v1.0 問題診斷（沿用 v2.0）

依現有實作截圖（船隊列表頁）檢視，問題並非「Kiro 做錯了」，而是**v1.0 文件本身給的規格太保守、太模糊**：

| 問題 | v1.0 文件的錯誤 | 修正方向 |
|---|---|---|
| 背景紋理完全看不到 | opacity 訂在 8-12%，在淺色 `--chart-paper` 底色上幾乎等於不存在 | 提高到 18-28%，並用更深的線色（非純裝飾灰,而是帶一點 `--ink-slate`飽和度) |
| 只給文字描述「SVG pattern 或 Canvas」，沒有具體規格 | 沒有告訴 Kiro 線條密度、鋪陳範圍、參考做法 | 直接附 SVG/CSS 範例程式碼與具體參數 |
| 全站看起來只有「一種平淡的質感」，沒有一個讓人記得住的焦點 | 過度強調「克制」，導致每個地方都很淡、沒有一個地方大方使用 | 新增「雷達/聲測環」作為限定頁面使用的大方視覺焦點 |
| 表格頁面資訊層級扁平（見圖1：標題、數字、標籤視覺權重幾乎相同) | v1.0 文件第 6 節提到但不夠具體 | 補上明確的字重/字級對照表 |

## 1. 參考來源一：quartermaster.us（雷達 / 等深線）

**明確聲明：以下是抽取設計原理，不是複製視覺樣式。** quartermaster.us 是深色底（navy）+ 白線，我們的主要工作區是淺色 `--chart-paper` 底（航海圖紙意象），所以線條的明暗需要反過來處理，但抓的原則相同：

| 抽取的原則 | quartermaster.us 的做法 | 我們的對應做法 |
|---|---|---|
| 等深線要「看得到」，不是心理安慰式的裝飾 | 白線在深底上 opacity 約 15-25%，肉眼一看就注意到紋理存在 | 深色系細線（`--ink-slate`）在 `--chart-paper` 淺底上，opacity 18-28% |
| 等深線鋪滿整個版面，形成環境感 | 背景紋理從頁首延伸到頁尾，不是侵在角落的小裝飾 | 全頁面背景層鋪滿，非局部貼花 |
| 雷達視覺只在少數頁面/情境出現，但出現時很完整、很大方 | Careers 頁面用整個右側區塊做雷達動畫 | 只在 1-2 個明確情境使用完整雷達視覺，但用的時候要做滿 |
| 視覺語言幾乎零填色，純線條/描邊 | 全站沒有實心色塊裝飾，只有線條與極簡文字標籤 | 延續既有的線條/描邊風格,不新增任何填色裝飾 |

## 2. 參考來源二：National Geographic《Into the Amazon》（篩選後採用）

《Into the Amazon》是一個**品牌形象/敘事型編輯網站**（scroll-based 敘事、沉浸式攝影、章節式全螢幕轉場、氛圍音效），核心目的是「說一個故事、引發情感共鳴」。我們的產品是**每天要被使用的維運決策工具**，核心目的是「快速找到需要關注的船、快速理解原因」。兩者目標不同，因此本節先明確篩選：大部分手法不適合直接搬過來，只有一個核心概念值得轉譯。

### 2.1 篩選結果：不採用的部分（附理由，避免日後重複提議）

| NatGeo 的手法 | 為什麼不適合本產品 |
|---|---|
| 全螢幕沉浸式攝影/影片背景 | 會壓縮資料呈現空間，且與我們「線條/描邊為主、零填色裝飾」的視覺語言衝突；船舶儀表板需要的是資訊密度，不是情感渲染 |
| 章節式全螢幕捲動轉場（scrolljacking） | 使用者需要快速在頁面間跳轉找資料，不能被強迫走完一段敘事才能到下一步；效率優先於敘事節奏 |
| 氛圍音效 | 辦公室/船上多人使用情境下，背景音效是干擾而非加分，VoiceBot 文件已定義好功能性語音回覆的範疇，不需要額外氛圍音效 |
| 大量情緒化編輯攝影排版與襯線大標 | 與既有字體系統（Display 窄體/Body/Mono）及「零填色線條」美學方向不合，換掉會讓整站風格分裂 |

### 2.2 篩選結果：值得轉譯採用的核心概念

**「跟隨一滴水，從源頭到海洋」的路徑敘事結構**——NatGeo 用一條實際的地理路徑（河流）串起每個資料點（物種、地點、故事），使用者滑動/捲動時，視角沿著路徑移動，途中的資訊隨位置浮現。

這個概念轉譯到船隊維運場景**非常合適**，因為我們也有一條「真實路徑」可以用：**船舶的實際航線**。目前 Speed Loss 頁與 Noon Report 頁是把資料呈現成表格與時間序列圖表,但同樣的資料其實也對應著一條真實航線,把兩者結合可以做出一個有意義（非裝飾性）的沉浸體驗，同時服務真實決策用途。

因此新增第 3 節「航次路徑敘事」規格。

## 3. 新增功能：航次路徑敘事（Voyage Path Narrative）

**這不是裝飾性動畫，是既有資料的另一種呈現方式，用於單船頁面的補充視圖，不取代既有表格/圖表。**

### 3.1 使用位置
- 位於單船詳情頁（`/vessels/:imo/speed-loss` 或 `/vessels/:imo/overview`）新增一個「航次回放」檢視模式切換選項，與既有的表格/圖表視圖並列，使用者可自由切換，預設仍是既有的表格/圖表視圖

### 3.2 互動方式
- 主視覺：該航次的實際航線地圖（沿用 MapLibre GL JS），船舶圖示沿航線路徑移動
- 下方或側邊有一條**時間軸拖曳滑桿**（重用 Speed Loss 頁面既有的日期滑桿元件，非新做一套），拖曳時地圖上的船舶圖示同步移動到對應位置
- 船舶移動經過的路徑上，**在有 Noon Report 記錄的座標點顯示標記**，點擊或拖到該點時，旁側浮現資料卡：當日速度、油耗、海況、是否為異常值——這是 NatGeo「路徑上浮現資訊」概念的直接轉譯,但顯示的是真實決策數據,不是故事文案
- 若該航次中有檢查/清洗/坐塢事件，在地圖路徑上以事件標記呈現（沿用第一份文件地圖規格中的事件標記語言）

### 3.3 與既有頁面的分工
| 檢視模式 | 用途 |
|---|---|
| 既有表格/圖表視圖（預設） | 精確數據分析、比較、篩選 |
| 航次路徑敘事（新增,選用） | 直覺理解「這艘船這趟航行實際經過哪裡、哪裡出現異常」，適合跟同事口頭討論或跟不熟數據的主管說明情境時使用 |

### 3.4 視覺規格
- 延續零填色線條美學：路徑線用細線 + 漸層方向指示（非粗色塊)
- 船舶移動動畫使用 `--motion-glide`（300-500ms 位移過渡,第一份文件既有 token)
- 不使用任何攝影/寫實圖像，僅使用既有的線條/圖示語言，確保與全站視覺一致

### 3.5 為什麼這個功能值得做（而不是為了炫技）
這個檢視模式解決一個實際問題：**Noon Report 表格資料很難讓人直覺理解「異常發生在航程的哪個階段」**（例如：是不是特定海域海況造成的、還是全程都異常）。把資料錶到真實地理路徑上，比純表格更容易看出模式，這正是 NatGeo 手法裡「路徑敘事」真正有效的地方——不是因為它好看，是因為地理路徑本身是這份資料的一個有意義維度。

## 4. 排版層級修正（針對圖1船隊列表頁的具體問題）

目前列表頁的問題是**所有文字視覺權重太接近**（標題、船名、數字、狀態標籤幾乎同一個字級與字重），修正規則：

| 元素 | 現況問題 | 修正規格 |
|---|---|---|
| 頁面標題（如「LIST-01」標籤） | 太小、太不起眼，沒有存在感 | 維持座標式標籤語意，但字級可再放大 10-20%，並確保 hover 有明確互動回饋 |
| 表格欄位標題（船名、IMO 等） | 與資料內文字重相同 | 欄位標題改用 Body 字體的**大寫 + 加寬字距**版本，與資料內文明確區分層級 |
| Speed Loss % 數字 | 目前與其他欄位視覺權重相同，但這是關鍵決策數字 | 改用 Mono 字體加粗，且依數值高低套用色彩（如 > 8% 用 `--signal-red`），讓眼睛第一時間被引導到需要關注的船 |
| 狀態標籤（航行中/靠泊中/錨泊中） | 純文字 + 小圓點，辨識度低 | 圓點加大,並確保三種狀態顏色對比明顯可辨（不能只靠顏色，需搭配圖示或文字本身已足夠明確) |

## 5. 等深線背景系統（v2 規格）

### 5.1 視覺規格
- 線條顏色：`var(--color-ink-slate)`，opacity **20%**（可依實測微調至 18-28% 區間，以「肉眼一看就感覺到紙面有紋理」為判斷標準,而非「幾乎看不到」）
- 線寬：1px（不隨螢幕縮放加粗）
- 線條樣式：不規則等高線（模擬海圖水深標示的自然曲線感），非機械式規則網格
- 鋪陳範圍：**全頁面背景，從最頂到最底**,用 `background-repeat` 或多層 SVG 拼接鋪滿，不可只出現在某個角落當小裝飾

### 5.2 實作方式

**方式 A：SVG pattern（建議優先採用，效能最好）**

```html
<svg width="0" height="0">
  <defs>
    <pattern id="bathymetric" width="400" height="400" patternUnits="userSpaceOnUse">
      <path d="M0,200 Q100,150 200,200 T400,200"
            fill="none" stroke="var(--color-ink-slate)"
            stroke-width="1" opacity="0.2" />
      <path d="M0,240 Q100,190 200,240 T400,240"
            fill="none" stroke="var(--color-ink-slate)"
            stroke-width="1" opacity="0.16" />
      <path d="M0,280 Q100,230 200,280 T400,280"
            fill="none" stroke="var(--color-ink-slate)"
            stroke-width="1" opacity="0.12" />
    </pattern>
  </defs>
</svg>
```

```css
.app-background-layer {
  position: fixed;
  inset: 0;
  z-index: 0;
  background: url(#bathymetric);
  pointer-events: none;
}
```

**方式 B：若需要更自然的等高線效果**，可用 D3 的 `d3-contour` 對一個隨機/固定的 noise 網格生成等高線路徑，一次性生成 SVG path 後直接靜態使用（不需要即時運算，生成好就存成固定 SVG，避免執行期效能負擔）。

**驗收判斷方式**：截圖後用滴管工具在空白背景區取色，應該能明確看到與 `--chart-paper` 有可辨識的線條差異，而非幾乎同色。

### 5.3 分層規則

```
Layer 0（最底）  等深線背景層：opacity 18-28%，全頁鋪滿，靜態或極緩慢平移
Layer 1          Chart Paper 底色層，完全遮蔽 Layer 0（資料卡片內部不可有線條滲出）
Layer 2          資料內容層
Layer 3          互動層
Layer 4（最上）  VoiceBot 疊層
```

## 6. Signature 焦點元素：雷達 / 聲測環（Radar Ring）

這個元素只在明確限定的頁面/情境使用，但使用時要完整、大方，成為該頁面的視覺焦點，而不是到處都放一點點。

### 6.1 使用範圍（明確限定，避免濫用）

| 使用情境 | 是否使用雷達視覺 | 說明 |
|---|---|---|
| 船隊總覽 Dashboard（`/`）背景/地圖上方裝飾區 | ✅ 使用 | 作為整個產品「即時監控船隊」的視覺主張，放在頁首或地圖區塊旁的留白處，尺寸要大、要看得到 |
| VoiceBot 聆聽中狀態 | ✅ 使用（已在 voicebot 文件定義的聲波環，屬同一視覺家族，語意統一） | 兩者用同一套同心圓語言,一個代表「持續監控」，一個代表「正在聆聽」 |
| Empty State（查無資料）插圖 | ✅ 可使用簡化版 | 呼應「尚在搜尋/監控中」的語意 |
| Speed Loss / 油耗歸因 / Noon Report 等資料密集頁面 | ❌ 不使用 | 這些頁面需要最大可讀性，雷達視覺會干擾圖表閱讀 |
| 船隊列表表格頁 | ❌ 不使用 | 表格本身資訊密度已高 |

### 6.2 視覺規格

- 同心圓：3-4 圈，`stroke: var(--color-ink-slate)`，opacity 遞減（外圈 15% → 內圈 30%）
- 掃描線：一條從圓心放射出去的線，帶漸層淡出尾巴（`var(--color-brass-amber)`），緩慢旋轉一圈，周期 **8-12 秒**（比 VoiceBot 聆聽中的聲波環更慢、更沉穩，因為這裡代表的是「背景常態監控」而非「主動回應中」）
- 船舶標記：用既有的三角形船舶圖示（呼應第一份文件地圖規格），但這裡是裝飾性/示意性標記，不需要對應真實座標，可用小型 tick 標記 + 等寬字體標籤（例如 `#IMO9786654`），強化「儀表讀數」質感
- 顏色：全部使用線條/文字，不使用任何填色色塊，維持全站零填色裝飾的原則

### 6.3 範例結構

```html
<svg viewBox="0 0 400 400" class="radar-signature" aria-hidden="true">
  <circle cx="200" cy="200" r="60" class="radar-ring radar-ring--1" />
  <circle cx="200" cy="200" r="120" class="radar-ring radar-ring--2" />
  <circle cx="200" cy="200" r="180" class="radar-ring radar-ring--3" />
  <line x1="200" y1="200" x2="200" y2="20" class="radar-sweep" />
  <!-- 船舶標記示意，可用既有地圖三角形圖示元件重用 -->
</svg>
```

```css
.radar-ring { fill: none; stroke: var(--color-ink-slate); stroke-width: 1; }
.radar-ring--1 { opacity: 0.3; }
.radar-ring--2 { opacity: 0.2; }
.radar-ring--3 { opacity: 0.15; }
.radar-sweep {
  stroke: var(--color-brass-amber);
  stroke-width: 1.5;
  transform-origin: 200px 200px;
  animation: radar-rotate 10s linear infinite;
}
@keyframes radar-rotate { to { transform: rotate(360deg); } }
```

### 6.4 與既有測深計（Fathometer）元件的分工

| 元素 | 語意 | 使用場合 |
|---|---|---|
| 測深計量表（既有） | **單一船隻/單一指標**的量化風險呈現（污損程度、維修急迫度） | 單船頁面、列表中的風險欄位 |
| 雷達/聲測環（新增） | **整個船隊/系統**的常態監控氛圍呈現，不對應具體數值 | 船隊總覽頁背景、VoiceBot 聆聽狀態、Empty State |
| 航次路徑敘事（第 3 節，新增） | **單一航次**的地理路徑資料回放 | 單船頁面的選用檢視模式 |

三者分工明確，不會在同一區塊互相搶視覺焦點。

## 7. 跨模組一致性更新（銜接 VoiceBot 文件）

- VoiceBot 文件中定義的「聲波環」與本文件的「雷達/聲測環」**必須是同一組視覺語言**（同心圓 + 線條 + 掃描動態），差異只在**速度**（聆聽中較快/主動，背景監控較慢/沉穩）與**觸發時機**
- Motion Token 補充一項：`--motion-radar-sweep`：8-12s, linear，專用於雷達掃描線旋轉
- 若使用者在 VoiceBot 模式詢問單船航次相關問題，淡入的區塊可以是第 3 節「航次路徑敘事」的精簡版（例如只顯示路徑+單一異常標記），與既有 voicebot 文件「淡入既有 Dashboard 元件」原則一致，不需為 VoiceBot 另做一套

## 8. 沉浸式粒子效果：夜光藻尾流（Bioluminescent Wake）

> 參考來源：luganolivinglab.ch 首頁的側邊局部粒子場（canvas/WebGL 生成的散點雲，非滿版）。
> 這一節直接回應你的兩個不滿意點：**配色制式化（navy+amber）**、**底色缺乏創意**。

### 8.1 先解決配色問題：拆分「功能色」與「氛圍色」

**問題根源**：目前 navy + brass-amber 之所以感覺制式，是因為這兩色被當成「全站唯一的視覺身份」在用——每個頁面、每個元件都在重複同一組顏色，久了就變成樣板感,而不是這兩色本身不好。

**解法：不整套換色（現有元件已大量使用這兩色，全面換色風險高且沒必要），而是新增一組「氛圍專屬色」，只用在裝飾性的粒子效果裡，絕不出現在按鈕、文字、狀態標籤等功能性 UI 上。** 這樣主色系統維持穩定（既有元件不用改），但產品的「第一印象」多了一個不落俗套的識別層。

| Token 名稱 | Hex | 用途限制 |
|---|---|---|
| `--atmo-bioluminescent` | `#5FE8B8` | **僅限**粒子效果本身，不可用於任何按鈕/文字/圖表/狀態色 |
| `--atmo-abyss-violet` | `#5B4B8A` | **僅限**粒子效果本身，同上限制 |

> 這兩色需補進第一份文件《Frontend Design Requirements》第 3 節 tokens 表，並明確標註「氛圍限定色，禁止用於功能性 UI」，避免日後被誤用到按鈕或狀態標籤上又變成第二套制式配色。

### 8.2 意象設定：「夜光藻尾流」而非直接畫船

你說「不用太直接」，所以不做船的剪影或貨櫃圖形，改用一個真實存在的海洋現象轉譯：**熱帶海域夜間航行時，船尾攪動夜光藻（bioluminescent plankton）留下的發光尾流**。粒子場本身不構成任何具體形狀，只是「密度與亮度呈現一種弧線擴散感」，比 Lugano 原本的隨機噴濺更有敘事意含,但視覺上依然抽象、不直白。

- 粒子分布：仿造 Lugano 的做法，**密度由一側向另一側遞減**（例如右側密集、往左側/內容區逐漸稀疊消散），形成類似船尾流擴散的視覺方向感
- 顏色：以 `--atmo-bioluminescent`（青綠）為主體粒子色，少量 `--atmo-abyss-violet`（深紫）點綴其中作為深水對比色，比例約 4:1，呼應 Lugano 雙色點狀分布的做法但改成海洋色調
- 粒子大小與亮度不均：多數為極小暗點（模擬遠處微弱發光），少數較大較亮（模擬近處攪動較劇烈處），避免均勻分布顯得機械

### 8.3 使用範圍（局部側邊，非滿版——比照 Lugano）

| 使用位置 | 是否使用 | 說明 |
|---|---|---|
| 系統登入/進入畫面（Splash） | ✅ 使用，作為主要識別時刻 | 比照 Lugano 首頁做法：標題置左、粒子場占右側 40-55% 版面，非滿版鋪滿 |
| 船隊總覽 Dashboard 頁首（KPI 卡片上方留白區） | ✅ 使用，尺寸較小 | 僅在頁首橫幅區域，不得延伸進 KPI 卡片或地圖區域 |
| 其他所有內頁（Speed Loss、油耗歸因、Noon Report 等） | ❌ 不使用 | 這是「入口/身份識別」層級的效果，不是全站背景系統;與第 5 節等深線背景（全站通用,克制)、第 6 節雷達（限定監控情境)屬於不同層級,三者不互相取代 |

**與既有兩個背景/氛圍元素的關係**：

| 元素 | 使用範圍 | 存在目的 |
|---|---|---|
| 等深線背景（第 5 節） | 全站所有頁面 | 環境質感，永遠存在的紙面紋理 |
| 雷達/聲測環（第 6 節） | 監控情境限定頁面 | 功能語意：代表「常態監控中」 |
| 夜光藻尾流粒子場（本節，新增） | 僅登入畫面 + Dashboard 頁首 | **品牌識別時刻**，只在使用者「剛進來」的那一刻出現，建立第一印象，之後進入操作情境就交還給前兩者 |

### 8.4 技術實作建議

- **優先用 Canvas 2D**，而非 WebGL/Three.js——粒子數量控制在 400-800 個即可達到 Lugano 的視覺密度感,Canvas 2D 效能已足夠,不需要引入 WebGL 的複雜度與心智負擔（這是一個內部維運工具,不是行銷網站，維護成本要納入考量）
- 粒子運動：極緩慢的漂移(參考 `--motion-ambient` token，20-40 秒尺度的緩慢變化)，不要做成快速閃爍或高頻噴發,維持沉穩、深海感,而非科技感的活躍粒子特效
- 響應式：手機版直接不渲染粒子層（純色底即可），避免行動裝置效能負擔與電量消耗
- 務必尊重 `prefers-reduced-motion`：降級為靜態粒子點陣（無漂移動畫）

### 8.5 為什麼登入畫面適合做這個「大方」的識別時刻

呼應第一份文件設計原則「把大膽留給一個地方」——目前全站的克制原則（等深線很淡、雷達限定使用）都是對的，但也代表**整個產品目前沒有一個「令人記住」的入口時刻**。登入畫面訪問頻率低、停留時間短、不承載即時決策資訊，是全站唯一可以「奢侈地」使用一個更具風格化、更飽和的視覺效果的地方，也最適合放這組專屬氛圍色，不會有干擾資料判讀的風險。
