目前專案裡其實有兩條並存、沒有互相呼叫的 API client 路徑，這是後續開發最容易搞混的地方：

路徑 B：直連層

FastAPI

backend-api/app.py → handler.py → DynamoDB
本機：localhost:8000／雲端：CloudFront

路徑 A 用在： 船隊總覽、船隊列表、Speed Loss、維修效能分析、跨船隊分析、船舶總覽卡片（都需要「整艘船的完整摘要」）。
路徑 B 用在： Noon Reports、水下檢查、油耗歸因、燃油預測（都是「單一 API 端點的原始資料」，不需要整合摘要）。
兩邊各自定義了一份幾乎重複的 TypeScript 型別（ApiVesselDetail vs BackendVesselDetail 等），維護時容易改一邊漏一邊。
環境變數行為不一致： api/client.ts 若沒設定 VITE_BACKEND_BASE_URL 會直接 throw；services/backend.ts 則靜默 fallback 成空字串（呼叫相對路徑）。目前 .env 設的是 http://localhost:8000，正式環境對應 CloudFront domain。

檔案功能資料串接狀態現況

§8 本輪修整記錄（2026-07-15b）

依 §7 清單執行，遇到跟 docs/main-requirement.md 主需求衝突時以主需求為主。過程中另外發現 2 個原始清單沒列到、但直接違反主需求的正確性 bug，優先修正。

發現的正確性 bug（比 §7 原清單更貼近主需求核心）

UWI（純檢查）被誤判為有養護效果： main-requirement.md 明講「資料中有一類事件屬於『純檢查、無實體介入』……聰明的模型應能辨識出這類事件不應帶來效能改善」。但 frontend/src/api/adapter.ts 有兩處違反這條規則：
  • EVENT_TYPE_MAP.UWI 曾對應到 'hull_cleaning'，導致 SpeedLoss.vue 的「清洗週期」切分（cleaningDays）把一次沒有實際清洗的檢查事件當成週期邊界，連帶影響污損趨勢回歸的擬合窗口與「比較上次清洗週期」功能。
  • CORRELATION_TYPE_MAP.UWI 曾對應到 'Propeller Polishing'，且 MaintenanceCorrelation.vue 的前後對比迴圈會對所有事件（含純檢查）計算「改善%」與星等評級，等於把一次沒有效果的檢查算成有效養護。
已修正為：新增 SpeedLossEvent['type'] 的 'inspection' 型別給 UWI 專用；效能分析迴圈對 event_type === 'UWI' 直接 continue 跳過。行為現在對齊後端 speed-loss-attribution 端點自己的 physical_intervention / category: 'inspection_only' 邏輯（handler.py 裡本來就正確處理了這個規則，問題只出在前端另外寫的 maintenance-correlation 分析邏輯沒有沿用）。附帶修正：cleaningDays 原本也漏了把 DD（乾塢）算進清洗週期邊界，一併補上（對應後端 HULL_CLEAN_TYPES）。
FuelPrediction.vue 不只沒掛路由，接上去也會壞： 補路由前先核對了 request/response 契約，發現 services/backend.ts 的 predictFuelConsumption() 送出的欄位（speed_kn／draft_fwd／draft_aft／cargo_on_board／wind_scale／sea_height，且不帶 noon_day）跟 handler.py 實際讀取的欄位（大寫 A 類欄位 AVG_SPEED／FORE_DRAFT／AFTER_DRAFT／WIND_SCALE／SEA_HEIGHT，沒有 cargo_on_board 對應欄位）完全不符，且回應型別假設的 counterfactual.slow_by_1kn_speed（降速情境）在真實回應裡根本不存在——後端只提供 counterfactual_uwc_pp（執行 UWC+PP 的節省估算）。實測會發生：POST 仍回 200（因為 vessel_id 有給、noon_day 非必填不會 400），但大部分 what-if 欄位因為欄位名不符被後端悄悄忽略、改用後端內建預設值；接著前端讀取 result.counterfactual.fuel_saving_mt 等欄位會是 undefined，在 .toFixed() 時直接 throw，被 catch 後顯示「預測失敗」。已修正 BackendFuelPredictionInput／BackendFuelPredictionResult 型別與 FuelPrediction.vue 的欄位對應，改用 noon_day（預設取該船最新一筆日報 vessel.dataDayMax）+ 大寫 override 欄位，並移除完全無效的「貨載量」輸入欄。
