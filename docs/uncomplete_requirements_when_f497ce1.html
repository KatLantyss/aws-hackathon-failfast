§2 A. 油耗預測模型

主需求要求前端呈現位置狀態
給定航行/環境條件，預測當日主機全速油耗 /vessels/:imo/fuel-prediction（FuelPrediction.vue）→ POST /api/v1/predict/fuel-consumption（XGBoost v3 Hybrid） ✅ 已呈現
反事實推論「執行 UWC/PP 的節能效益」 同一頁面的 KPI 卡「執行 UWC+PP 可省油耗」「年化節省金額估計」，直接使用後端 counterfactual_uwc_pp 欄位 ✅ 已呈現
理解各類養護事件對運行/油耗的不同影響程度 /vessels/:imo/fuel-attribution（速損歸因：依 hull/propeller/hull+propeller/inspection_only 分類平均改善%）＋ /vessels/:imo/maintenance-correlation（維修類型 vs 平均改善效果長條圖） 🟡 部分呈現 — 透過速損歸因呈現，非透過油耗預測模型本身
效能如何隨時間退化、養護後如何恢復 Speed Loss 頁的污損趨勢回歸線＋清洗週期比較；maintenance-correlation 的時序圖（COR-01） 🟡 部分呈現 — 呈現的是「原始資料的趨勢」，不是「模型學到的退化/恢復曲線」，兩者不是同一件事
只給部分可見特徵，推估被遮蔽的油耗數值 無 ❌ 未呈現
客觀評分（模型準確度：MAE/RMSE/R²、predicted vs actual 對比） 無 ❌ 未呈現
「遮蔽特徵推估」目前只存在於離線腳本： backend-api/predict_submit.py 讀取 vt_fd.csv 裡標記為 PREDICT 的列（S21–S23 的待預測資料），逐筆呼叫 /api/v1/predict/fuel-consumption，結果限幅到合理區間（30–120 MT/day）後寫出 backend-api/submission.csv（比賽提交格式，如 S22,927,ME_FULLSPEED_CONSUMP_HSHFO,30.0）。這是離線評分流程，前端完全沒有讀取或視覺化這份 submission 結果（全站搜尋 submission／predict_submit 均無關聯命中）。
沒有任何模型準確度展示： 搜尋整個 frontend/src（mae／rmse／r2／accuracy／validation／score／predicted vs actual 等關鍵字）沒有找到任何有意義的命中。FuelPrediction.vue 只顯示單筆點預測值，看不出這個 XGBoost 模型在訓練集（S1–S12）上預測得準不準。
額外發現（非缺口，但值得記錄）： vessel.type（'training'/'prediction'）雖然在 VesselLayout.vue 與 VesselOverview.vue 有 render 出來，但 adapter.ts 對 training 船會把它改顯示成 "W1"/"W2"（航線分組），只有 prediction 船會顯示英文原字 "prediction"。兩邊都沒有清楚的中文「訓練集／預測集」標示，使用者不容易一眼看出哪些船是有真實油耗答案的訓練船、哪些是本題要預測的船。

§4 小結

已做到的部分相當完整： ISO 19030 的 ①②③ 三項全部有對應頁面，且都是打真實 API；UWI 誤判的正確性 bug 也已修正。油耗預測模型的「單日預測 + UWC/PP 反事實」核心功能也已修好並掛上路由（/vessels/:imo/fuel-prediction）。

兩個明確缺口 — 都跟「模型本身的客觀評分」有關，不是 Speed Loss Dashboard 那邊

  • 缺口 1 · 沒有模型準確度展示 沒有任何頁面告訴使用者這個 XGBoost 模型預測得準不準——MAE/RMSE/R² 或「預測 vs 實際」對比圖都沒有。目前 FuelPrediction.vue 只能看到單筆點預測，無法評估模型品質。
  • 缺口 2 · 沒有「遮蔽特徵推估」的示範 這件事目前只活在 backend-api/predict_submit.py 這支離線腳本裡，產出 submission.csv；前端從未讀取或視覺化這個流程與結果。
建議方向（尚未執行，待決策）： 要補這兩塊，大概需要後端新增一支端點（例如回傳訓練集上的驗證指標，或讓 submission.csv 的結果可查詢），前端再新增一個「模型評估」頁面去顯示——這是目前完全不存在的新功能，不是既有頁面的小修，需要額外規劃。
產出方式：基於同一次對話中對 docs/main-requirement.md 與 frontend/backend-api 程式碼的逐項比對與關鍵字搜尋（mae|rmse|r2|accuracy|score|validation、submission|predict_submit、vessel.type render 位置），未執行任何網路請求。對照的程式碼狀態為 git commit f497ce1。
